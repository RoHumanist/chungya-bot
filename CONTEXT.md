# 청약알리미 — AI Context Document

---

## ① 목적 (Why)

**한 줄:** 내 조건 한번 입력하면, 전국 청약 중 나에게 맞는 것만 알아서 알려주는 서비스.

**이걸 왜 만드는가:**
- 청약 정보는 청약홈(applyhome.co.kr)에 다 있지만, **까먹고 / 매번 확인하기 귀찮고 / 내 조건에 맞는지 모름**
- 기존에 이걸 해주는 서비스가 토스 앱인토스에 없음
- 핵심 가치: **탐색/확인 노동을 없애는 것**. 유저는 조건만 한번 입력하면 끝.

**이걸 안 하면:**
- 유저가 매일 청약홈 들어가서 새 공고 확인해야 함
- 조건 일일이 대조해야 함 (무주택? 소득? 통장? 나이?)
- 놓치면 끝 — 청약은 기간이 있으니까

## ② 현재 상태 (AS-IS)

**완성된 것:**
- Next.js 15 + Tailwind CSS 웹앱
- 토스 스타일 UI (모바일 퍼스트, 앱인토스 WebView 타겟)
- **온보딩**: 스텝별 한 화면씩 (나이 → 혼인 → 주택 → 통장 → 소득 → 관심유형 → 지역)
- **조건 매칭 피드**: 유저 프로필 기반 필터링, 지원가능/불가능 판별
- **청약 상세 페이지**: 분양가, 일정(D-day + 상태뱃지), 자격조건, 유형별 전략 가이드, 청약홈 링크
- **설정 페이지**: 조건 수정, 전국 지역 선택 (서울→구, 경기→시 자동 펼침)
- **DevTools**: 개발환경에서만 보이는 초기화/디버그 버튼
- Vercel 배포: https://chungya-bot-dgqy.vercel.app/
- GitHub: https://github.com/RoHumanist/chungya-bot

**데이터:**
- 현재 mock 데이터 (4건) — API 키 대기 중
- 공공데이터포털 "한국부동산원_청약홈 분양정보 조회 서비스" API 사용 예정
- 전국 데이터 지원 (서울만이 아님)

**블로커:**
- 공공데이터 API 키: 발급 대기
- 사업자 등록: 토스 SDK 접근에 필요
- 토스 SDK (`@apps-in-toss/web-framework`): 사업자 등록 후 접근 가능

## ③ 목표 상태 (TO-BE)

**MVP (1차):**
> 유저가 조건 입력 → 맞는 청약 알림이 옴 → 상세 확인 → 청약홈에서 바로 신청

이것만 되면 가치 있음. 출시만 하면 반응은 있을 것 (앱인토스에 청약 서비스 없음).

**2차 (수요 확인 후):**
- 시세차익 계산 (국토부 실거래가 API 연동)
- 주변 시세, 아파트 평가, 커뮤니티 분위기
- 경쟁률 표시 (2번 API: 청약접수 경쟁률 조회)
- 자금 조달 가이드

**성공 기준:**
- 유저가 한번 조건 넣고 → 알림 받고 → 청약 신청까지 이 앱 안에서 완결
- "이거 없으면 불편하다" 느낄 정도

## ④ 구조 (Structure)

### 기술 스택
- **프론트**: Next.js 15 (App Router) + Tailwind CSS + SWR
- **배포**: Vercel (GitHub 연동, push 시 자동 배포)
- **데이터**: 공공데이터포털 API → Next.js API Route → SWR → 클라이언트
- **상태**: React Context (ProfileProvider) + localStorage
- **타겟 플랫폼**: 토스 앱인토스 WebView (현재는 웹으로 운영)

### 디렉토리 구조
```
src/
├── app/                    # Next.js App Router 페이지
│   ├── page.tsx            # 홈 (온보딩 or 매칭 피드)
│   ├── providers.tsx       # Context Provider 래퍼
│   ├── subscriptions/      # 청약 목록 + 상세
│   │   ├── page.tsx        # 전체 목록 (필터 칩)
│   │   └── [id]/page.tsx   # 상세 (일정, 조건, 전략, 신청버튼)
│   ├── settings/page.tsx   # 내 조건 수정
│   └── api/subscriptions/  # API Route (mock ↔ 실데이터 전환점)
├── components/
│   ├── Onboarding.tsx      # 스텝별 온보딩 (8스텝)
│   ├── MatchedFeed.tsx     # 조건 매칭된 피드
│   ├── SubscriptionCard.tsx # 청약 카드
│   ├── ProfileForm.tsx     # 조건 수정 폼
│   ├── BottomNav.tsx       # 하단 네비게이션
│   ├── LoadingFeed.tsx     # 스켈레톤 로딩
│   └── DevTools.tsx        # 개발용 도구 (production에서 숨김)
├── lib/
│   ├── filter-context.tsx  # ProfileProvider (전역 상태)
│   ├── match.ts            # 매칭 로직 (순수 함수)
│   ├── api-client.ts       # 공공데이터 API 호출
│   ├── transform.ts        # API 응답 → 앱 타입 변환
│   ├── mock-data.ts        # 테스트용 mock 데이터
│   ├── use-subscriptions.ts # SWR 훅
│   └── format.ts           # 가격/면적/날짜 포맷
└── types/
    └── subscription.ts     # 타입 정의 + DEFAULT_PROFILE
```

### 데이터 흐름
```
[공공데이터 API] → api/subscriptions/route.ts → transform → Subscription[]
                                                                    ↓
[localStorage] → ProfileProvider → UserProfile ──→ match.ts → MatchResult[]
                                                                    ↓
                                                         MatchedFeed / 목록 렌더링
```

### 핵심 타입
- `Subscription`: 청약 정보 (이름, 유형, 지역, 세대, 분양가, 일정, 조건)
- `UserProfile`: 유저 조건 (나이, 혼인, 주택, 통장, 소득, 관심지역/유형)
- `MatchResult`: { sub, eligible, reasons[] } — 매칭 결과

## ⑤ 규칙 (Rules)

### 매칭 로직 (`match.ts`)
- 지역: `__ALL__` = 전국, `[]` = 미선택(전체), `["서울 강남구"]` = 특정 지역
- 유형: `[]` = 전체, `["민간분양", "공공분양"]` = 선택한 것만
- 자격: 무주택/세대주/나이/소득/자산/통장 조건 하나라도 안 맞으면 `reasons`에 추가
- `eligible = reasons.length === 0`

### 지역 선택 규칙
- 서울/경기/인천은 하위 지역(구/시) 선택 가능
- 나머지 광역시도는 시·도 단위 토글
- "전국 전체" 누르면 `["__ALL__"]` 저장, 다시 누르면 `[]`로 해제
- 전국 선택 시 하위 지역 목록 숨김

### 청약 유형 (8종)
민간분양, 공공분양, 신혼희망타운, 청년특별공급, 행복주택, 국민임대, 장기전세, 오피스텔

### API 전환
- `PUBLIC_DATA_API_KEY` 환경변수가 있으면 실데이터, 없으면 mock
- API route (`/api/subscriptions`)가 유일한 전환점

## ⑦ 예시 (유저 플로우)

**신규 유저:**
1. 앱 진입 → 온보딩 ("반가워요!" 화면)
2. 나이 30세, 미혼, 무주택, 통장 24개월... 차례로 입력
3. 관심 유형: 청년특별공급, 민간분양
4. 관심 지역: 서울 강남구, 서울 송파구
5. "청약 보러가기" → 매칭 피드
6. 카드 클릭 → 상세 (일정, 조건, 전략)
7. "청약홈에서 신청하기" → 청약홈 이동

**기존 유저:**
1. 앱 진입 → 바로 매칭 피드 (온보딩 스킵)
2. 새 청약 확인 → 상세 → 신청
3. 조건 바뀌면 "내 조건" 탭에서 수정

## ⑧ 제약/주의사항

- **Accuracy는 중요하지 않음** — 청약 정보 자체는 청약홈에 다 있음. 우리의 가치는 "필터링 + 알림"
- **과도한 기능 추가 금지** — MVP는 "맞는 청약 알림". 시세차익/커뮤니티는 수요 확인 후
- **사업자 등록 없으면 토스 출시 불가** — 웹 버전으로 먼저 운영 가능
- **TDS(토스 디자인 시스템) 필수** — 앱인토스 검수 기준에 포함
- **API 호출 제한** — 공공데이터포털 일 1,000건
- **localStorage에 민감 정보** — 소득/자산 저장됨. 암호화 없음. 현재는 허용 범위.

## ⑨ 피드백 기준

- **유저가 3분 안에 온보딩 끝내고 피드를 볼 수 있는가?** → OK/수정
- **내 조건에 안 맞는 청약이 피드에 뜨는가?** → 버그
- **청약 상세에서 "이걸로 넣어야겠다" 판단할 수 있는 정보가 충분한가?** → 부족하면 추가
- **청약홈까지 2탭 이내로 갈 수 있는가?** → UX 기준

## ⑩ 우선순위 기준

1. **유저 가치 직결** → 매칭 정확도, 알림, 청약 신청 연결

3. **UX 개선** → 디자인, 로딩, 에러 처리
4. **추가 기능** → 시세차익, 경쟁률, 커뮤니티 (수요 확인 후)
